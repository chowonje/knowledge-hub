## What this project is

`knowledge-hub` is a local-first, policy-first, retrieval-assistant-first knowledge runtime.

## Current public release posture

- Public-facing posture is **Research Preview**, not stable-release language.
- The supported default path for public docs is `discover -> index -> search/ask -> evidence review`.

## Current representative scenario

The current product story is intentionally narrower than the full implementation surface.

- Primary user: a solo researcher or builder working against a local corpus.
- Primary job: collect papers/web/vault notes locally, index them, ask grounded questions, and inspect the evidence trail before reusing the output.
- Current non-goal for the default surface: multi-step research automation, deep synthesis, or broad operator workflows as the first product promise.

The representative default loop is:

`discover -> index -> search/ask -> evidence review`

Current success criteria for that loop:

- `discover`: at least one source item lands in canonical local storage.
- `index`: the retrieval-facing vector/document surface is non-empty and readable by the runtime.
- `search/ask`: the runtime returns either a grounded result or an explicit lack-of-evidence outcome.
- `evidence review`: the answer/result exposes enough source trace to inspect the grounding path.

The answer path now makes that evidence contract explicit in runtime payloads. `khub ask` carries additive `EvidencePacket`, `AnswerContract`, and `VerificationVerdict` objects that pin source ids, real source-content hashes, span offsets, citation refs, pass/fail/abstain verdicts, and conservative-fallback state. In strict mode, spans without canonical source hash/offset provenance or spans flagged stale are excluded from evidence rather than treated as citation-grade grounding; snippet hashes stay separate from source-content hashes. These objects are registered in the JSON schema validator and are intentionally additive so older top-level answer payload fields remain compatible.

Verification is treated as a gate, not a claim-repair loop. Unsupported claims, weak claims with no supported claim, uncertain-only answers, and caution/missing-conflict-language cases are routed to conservative fallback instead of LLM rewrite. `VerificationVerdict.rewriteAllowed` opens only for `pass`; missing policy classification is `UNKNOWN` and fails closed for external model use. ask-v2 card evidence with weak/missing verification plus unsupported fields now marks the packet `askV2HardGate` and returns a no-result/lack-of-evidence answer before generation.

The Opus architecture-hardening replay has a clean sibling integration branch: `opus-architecture-hardening-clean-20260423` in `../knowledge-hub-opus-hardening-clean`. That branch is not just the standalone `12` overlay; it is the dependency-closed replay of checkpoint buckets `01`-`07` plus `00`, `09`, `10`, `11`, and `12`. The clean replay now passes the full Python suite (`885 passed, 5 warnings`), the affected Python suite (`287 passed`), Foundry `npm run check`, Foundry `npm test` (`56 passed`), runtime `py_compile`, and staged diff hygiene (`git diff --check --cached`). The final whitespace pass only normalized line endings/trailing spaces in replayed docs/eval artifacts and older carryover files. The follow-up split branch `opus-hardening-split-20260424` carries the same stack as reviewable commits plus a release-hygiene cleanup commit that passes public-release hygiene and release smoke locally.

The split branch now also has a shared Python/Foundry policy conformance fixture at `docs/policy/policy-conformance-cases.json`. Python `evaluate_policy_gate` and Foundry `LocalPolicyEngine` both consume that fixture, covering declared P0, nested `policy.classification=P0`, detected P0, structured P1 JSON content, summary-shaped P2, and public-url P3 behavior. Foundry treats payload-declared classification as advisory input and combines it with detected payload sensitivity, keeping Python as the policy-authority reference while TypeScript remains an enforcer/runtime boundary.

Additive subsystems exist behind that default surface:
- ontology / claims / event history
- document-memory / paper-memory / claim-normalization / synthesis
- Foundry-style agent-runtime delegation through `foundry-core`

## Product shape

The default product shape is now intentionally split into three layers:

- `Core Runtime`: the local grounded loop (`discover -> index -> search/ask -> evidence review`) plus read-only task-context assembly and policy/provenance surfaces
- `Agent Gateway`: a narrow future-facing adapter layer for context packs, grounded tool access, approval/policy checks, and result receipts
- `Labs`: experimental or operator-heavy surfaces that remain additive and do not redefine the default promise

This means two things operationally:

- the project should keep narrowing around the repeatable local grounded loop rather than broadening the default product surface
- agentic / AGI alignment remains in scope, but through a narrow gateway rather than by turning the whole product into a general agent platform

## Current architecture

### Python runtime surface (`knowledge_hub/`)
- Python remains the default product/runtime for local data access, persistence, indexing, retrieval, answer orchestration, task-context assembly, and CLI/MCP exposure.
- Canonical ownership is now explicit:
  - `knowledge_hub.interfaces.*` for CLI/MCP entrypoints and command registration
  - `knowledge_hub.application.*` for runtime composition and Foundry bridging
  - `knowledge_hub.infrastructure.*` for config/persistence/provider surfaces
  - `knowledge_hub.core.*` for low-level local primitives and compatibility exports
  - `knowledge_hub.domain.*` for source/domain-specific rules that sit above the generic engine
- Config, vector DB, repository contracts, ontology profile internals, SQLite store implementations, paper lane taxonomy, and section-card persistence now use `infrastructure.*` or feature-local canonical modules, while legacy `core.*` paths remain compatibility shims for tests and external callers.
- Legacy `knowledge_hub.cli.*` and `knowledge_hub.mcp_server` are compatibility shims only; internal package code should not import them directly.
- `knowledge_hub/domain` is no longer just a model alias bucket. It now contains registry/protocol scaffolding plus concrete domain packs such as `knowledge_hub.domain.ai_papers` and `knowledge_hub.domain.web_knowledge`.
- `knowledge_hub/ai` now trends toward a generic engine surface. Paper-specific routing/query/claim rules are moving under `domain.ai_papers`, while `knowledge_hub.ai.paper_query_plan` and `knowledge_hub.ai.claim_cards` stay as compatibility shims for one stabilization cycle.

### TypeScript runtime surface (`foundry-core/`)
- `foundry-core` owns the stricter agent-runtime abstraction, policy/audit contracts, and bridge-facing execution model.
- When `foundry-core` delegates back into Python, the canonical entrypoint is `knowledge_hub.interfaces.cli.main`.
- `foundry-core` is the preferred delegated runtime for agent execution, not the sole runtime path.

### Integration boundary
- Python is still the default end-to-end runtime for the product.
- Agent execution can delegate into `foundry-core`, with Python fallback when the bridge is unavailable.
- The migration direction is additive: tighten contracts without forcing a disruptive runtime flip.
- Python remains the final authority for task-mode classification, policy gating, and normalized run payloads.
- Authority hardening is now explicit: Python owns `knowledge.db`, Chroma, source stores, and product-facing outputs; TypeScript owns delegated execution flow plus `.khub/personal-foundry/*` state; bridge transport remains subprocess + JSON stdout with shared P0 detection inputs and shared bridge fixtures guarding drift.
- Agent run artifacts returned from Foundry are re-evaluated by the Python policy gate. Declared artifact classification is advisory: if `jsonContent` or nested artifact payloads match P0 detection, the normalized run is blocked, the artifact body is replaced with the policy redaction marker, and writeback is denied.
- Foundry also self-protects direct TypeScript runtime consumers: `agent_verify` denies detected P0, declared non-P0 classifications are treated as advisory against payload re-scan, and verify-blocked P0 artifacts are redacted in the runtime envelope before return. Python remains the product-facing final authority and also upgrades non-P0 artifacts to the most sensitive detected class.

## Current default contract

- Default CLI/MCP discovery stays retrieval-assistant-first: ingest, search, ask, task-context, vault, health.
- Advanced workflows stay behind `khub labs ...` or profile-gated MCP discovery.
- Some additive compatibility commands still exist at the root CLI for direct invocation, but the default top-level help now hides personal/eval/maintenance surfaces such as `dinger`, `os`, `eval`, `paper-memory`, `math-memory`, and `vector-*`.

Foundry operator subcommands such as `sync`, `discover*`, and `foundry-conflict-*` remain directly invokable for compatibility but are hidden from the default `khub agent --help` output.
- The canonical foundry/operator CLI surface now lives under `khub labs foundry`. Legacy `khub agent sync|discover|discover-validate|foundry-conflict-*` paths remain as hidden compatibility aliases, but the intended split is now explicit in the command tree instead of existing only as hidden help policy.
- That split now also exists at the implementation boundary: `knowledge_hub.interfaces.cli.commands.foundry_cmd` owns the foundry/operator command callbacks plus sync/discover helper stack, while `knowledge_hub.interfaces.cli.commands.agent_cmd` is back to gateway-facing commands and alias registration only.
- The module boundary is now test-protected: architecture and CLI regressions check that `agent_cmd.py` does not re-define foundry/operator implementations, `foundry_cmd.py` does not import or self-register `agent_group`, the labs lazy-load path points at `foundry_cmd`, and hidden legacy aliases remain directly invokable.

- Large A/B experiment artifacts now have an explicit outside-the-repo landing zone. The first physical split moves bulky `runs/ab/*` outputs toward `~/.khub/runs/ab`, while repo-local `runs/ab` can remain as a compatibility symlink so existing docs and scripts keep working without leaving cloned repos and virtualenvs inside the product checkout.

The default local landing zone for ignored `eval/knowledgeos/runs/*` outputs is `~/.khub/eval/knowledgeos/runs`, while repo-local `eval/knowledgeos/runs` can remain a compatibility symlink so existing eval scripts, reports, and docs keep working without storing machine-eval payloads inside the checkout.
- Answer-loop now has an explicit internal post-collect seam: `collect` still runs inside the Python core path, but the downstream `judge -> summarize -> autofix` lane is routed through a small executor adapter so future eval-satellite extraction can swap that lane without rewriting `run_answer_loop()`. When the loop needs config-backed CLI replay, the subprocess path now uses the canonical `khub labs eval answer-loop ...` command family instead of the hidden root compatibility alias, including `autofix`.
- Inside that lane, the patch engine is now also isolated one step further: `autofix_answer_loop()` still owns manifest reading, failure-card generation, and patch-brief assembly, but the actual Codex patch execution plus immediate diff/verification checks now sit behind a dedicated patch executor. This narrows the remaining extraction problem to “where should the patch runner live” instead of “how does the whole autofix workflow work”.
- Those executor seams are now test-protected at the architecture boundary: regressions fail if `run_answer_loop()` directly re-couples to judge/summarize/autofix functions or if `autofix_answer_loop()` reabsorbs Codex patch execution, git diff inspection, or targeted verification instead of routing through `AnswerLoopPatchExecutor`.
- Default `paper` ask behavior is now intentionally narrow and predictable: `concept_explainer`, `paper_lookup`, `paper_compare`, and `paper_discover` are the only default paper families.
- For `concept_explainer`, representative anchors may guide ranking and context shape, but they do not single-scope the evidence set unless the evidence policy explicitly requires single-scope behavior. Support papers remain available for concept-first explanations.
- Heavy multi-paper synthesis is not part of the default paper contract; it stays on `khub labs paper topic-synthesize`.
- Domain routing is intentionally simple in this phase: `source=paper` resolves to `ai_papers`, `source=web` resolves to `web_knowledge`, `source=youtube` resolves to `youtube_knowledge`, and `source=vault` resolves to `vault_knowledge`.
- Repo/project context is inspectable and ephemeral; it does not persist into SQLite, vector DB, or ontology tables.
- Local-first privacy and outbound policy gating are hard constraints.
- Prefer additive migration over disruptive redesign; preserve existing CLI/MCP contracts unless promotion is explicit and verified.
- Current stabilization focus is `contract + entrypoints + health/integrity`; `paper/public reading`, `retrieval/ask-v2`, `learning/web/vault`, and `foundry/satellites` remain separate hardening buckets rather than part of this green-baseline slice.

- The replay plan is now explicit at the tranche level: `00-replay-control-plane` captures the split/replay helper surface, `09-core-loop-stabilization` stays frozen to the clean sibling `knowledge-hub-core-loop-stabilization-head-v11`, and `11-weekly-ask-ci-release-hygiene` is the later overlay that promotes the weekly smoke to `6/6` and narrows required PR CI.

- Answer-loop compare packets now also carry evidence-use classes for supporting sources. For `paper + comparison` rows, supporting evidence is tagged as `direct_comparative_evidence`, `background_evidence`, `task_specific_example`, or `weak_indirect_evidence`, and the compare prompt explicitly restricts the main difference claim to anchor/direct/background evidence while keeping task-specific examples in the example/limitations lane.
- Answer-loop compare prompts no longer embed CNN/ViT-specific section labels. The response frame is now generic across compare targets, so `BERT vs GPT` and similar non-vision comparisons reuse the same axis-first packet contract without leaking old target names into the answer body.
- Answer-loop generation prompts now also require the backend to match the user's question language/register unless the packet says otherwise. This does not fully forbid borrowed paper titles or evidence labels, but it reduces mixed-language answer framing on Korean compare/explainer queries and keeps the behavior inside the existing frozen-packet generation layer.
- Answer-loop compare packets now materialize an additive `axisEvidenceMatrix` before answer generation. For each fixed compare axis, the matrix records which target-anchor sources directly support each side, which supporting sources are only shared/background evidence, and whether the current packet has `supported | partial | insufficient` coverage on that axis. This is still lighter than full claim-level grounding, but it moves the loop from a flat evidence bundle toward a structured compare-table assembly step.
- Answer-loop collect now also performs a narrow compare-answer verification pass against that matrix. The current v1 check is intentionally lightweight: if an answer explicitly discusses a low-coverage axis such as compute/scaling without also acknowledging insufficiency, the collected row is marked `axis_gap_detected`; otherwise it records `axis_matrix_checked` with the acknowledged low-coverage axes. This still is not sentence-level claim verification, but it gives the eval loop an inspectable bridge between structured packet evidence and the final answer text.
- That compare verification is now more inspectable at the answer-text level: the verifier splits the final answer into lightweight sentence/table-row segments, links low-coverage-axis mentions back to those segments, and records short unsupported snippets in the verification result. This is still not full claim provenance, but it is the first concrete step from axis-level warnings toward sentence-level grounding checks.
- The answer-loop Codex path now treats Codex as a server/backend over a CLI-compatible transport by default. Answer generation uses `codex exec` with a read-only sandbox, patching uses `codex exec` with workspace-write, and direct MCP remains opt-in through `KHUB_CODEX_TRANSPORT=mcp` for environments whose client stack can consume Codex event notifications without validation failures.
- Answer-loop packet building no longer bypasses the paper-aware retrieval planner. Frozen packet collection now runs through `RetrievalPipelineService` with domain-normalized query context when available, so concept-explainer paper queries can carry representative paper hints and resolved-source promotion into the eval packet instead of comparing backends on a degraded raw-search ranking.
- Answer-loop answer formatting now also avoids fake inline citations in plain-text outputs. Packet evidence context is rendered as `source N:` instead of `[N]`, the backend prompt explicitly forbids bracketed reference markers, and collected answer text strips leftover `[1]` / `(Source 1)`-style tokens because the eval CSV does not render them as real citations.
- GPT-5 family OpenAI generation now goes through the Responses API in the provider layer. In practice this matters because `gpt-5-mini` was returning empty strings through the old Chat Completions path under the answer-loop workload (`finish_reason=length` with no visible content), while the Responses path with minimal reasoning returns stable visible text for the same prompts.
- Outbound policy classification for answer-loop packet summaries is now less brittle against retrieval-score formatting. Packet context rounds scores before policy inspection so a long decimal tail cannot accidentally resemble a blocked long-number secret/card pattern and falsely stop an otherwise safe external answer-generation call.
- The answer-loop evaluator assets now include a reusable system-diagnosis prompt and rubric for side-by-side answer review. Those assets are intended to explain whether a compare-answer failure is dominated by `MODEL`, `RETRIEVAL`, `ASSEMBLY`, or `PROMPT` without widening the machine-judge CSV schema beyond the existing `pred_*` columns.
- The automatic loop score is now explicit: `pred_label_score` is the primary comparator, groundedness/source-accuracy/abstention agreement are guardrails, and the loop stops on max attempts, lack of improvement, missing judged rows, dirty-worktree policy blocks, patch failure, or targeted verification failure.
- `khub labs eval answer-loop optimize` now adds a separate offline-only optimization lane that does not mutate code or retrieval. It reuses the frozen-packet collect surface, forces both generation and judging through the Codex transport, estimates token usage from prompt/output length because Codex does not expose live usage counts, caps judge usage to a configured fraction of the daily estimated-token budget, and writes a review pack plus holdout-audited best-answer bundle for later human or external-model review instead of running `autofix`.
- The optimizer now treats `implementation_steps` rows as a stricter answer style. Generation prompts ask for concrete repo-navigation steps and explicit uncertainty when the packet does not confirm an exact file or symbol, the private judge penalizes generic architecture prose for that style, and the early-success gate now requires usefulness to be good for implementation-style packets so a holdout-only downgrade is less likely.
- Claim-first population now has an operator-facing mixed rollout for papers. Broad coverage continues to use the existing `paper sync-keywords --claims -> claims normalize --all -> labs claim-cards backfill` path, while strict comparison work can now re-extract exact paper subsets through `khub claims extract-paper --paper-id ...|--paper-id-file ...` and generate bounded strict-lane artifacts with `khub claims strict-report --out-dir ...`. The strict report reuses stored ClaimCards, normalization rows, alignment refs, pending claims, and stage-1 verification to emit `strict_lane_report.json` plus `strict_review_queue.csv` without changing ask/search behavior.
- Paper ClaimCard backfill is now resilient to late claim population. If a persisted `paper-card-v2` exists without `paper_card_claim_refs_v2` but the live DB already has accepted paper claims, `khub labs claim-cards backfill --source paper` first repairs the paper card via `PaperCardV2Builder`, then rematerializes ClaimCards and removes stale fallback ClaimCards that used to remain orphaned after the source card switched from fallback-only to extracted claims. This keeps broad-lane evaluation from reporting `syntheticFallbackRate=1.0` just because claim refs were never refreshed.

- The external `khub notebook` CLI group, matching MCP notebook-provider tools, provider adapters, and sync-manifest persistence layer have been removed from the maintained runtime.

- The repository enforces outbound policy checks before OpenAI or OpenAI-compatible provider calls.
- The default `khub` CLI now summarizes provider outbound warning records into a short stderr line with `classification`, `provider`, `model`, and `trace_id`, while `--verbose` preserves the raw `warnings=[...]` record for audit/debug use.

### Ontology and events
- Ontology entities, claims, aliases, and event-sourcing infrastructure exist.
- Event records and policy classification are part of the current design direction.

### Post-close-out operator workflow

- This workflow is follow-up guidance after close-out, not an expansion of the closed verified path. The public CLI already exposes `capture status|list|show|cleanup|requeue|retry`, but only the targeted behavioral slices stay in the frozen approval gate.
- `khub os project evidence` is the read-only candidate review surface above that capture path. It derives `projectEvidence.evidenceCandidates` from existing OS/source-ref state, but it is not itself a promotion or finalize command. The default text output now makes each candidate scannable in-place by surfacing `inbox id`, Dinger page, supporting refs, `show|explain` drill-down shortcuts, and data-derived attention hints on the list surface.
- That same list surface is now also an exploration surface over the existing read model rather than only a flat dump. It adds read-only filtering by `replayAction`, `inbox state`, and `source type`, recency-first ordering plus candidate `age/updated` hints, and a derived `recent reviewed` section built from existing resolved Dinger-backed inbox items. The newest additive scan layer derives candidate `attentionLevel/attentionBasis` only from current open-vs-resolved match counts, supporting refs, source-type breadth, and updated age, while recent-reviewed rows derive `revisitLevel` only from whether a related open candidate still exists. This remains additive operator guidance only: there is still no review receipt store, so the recent-reviewed section can truthfully preserve only inbox `resolved` state plus context/recency, not a persisted approve-vs-dismiss decision.
- `khub os evidence show` is the read-only detail surface for one candidate. It exposes the operator-grade explanation fields already present in the candidate reason: `why reused / why replayed`, matched open/resolved item summary, source refs, and the next review commands.
- `khub os evidence review` is now the thin operator action surface above that read model. It resolves a candidate by existing `inboxId` or Dinger note path, keeps `explain` as the read-only review path, and reuses existing OS verbs for state changes instead of introducing a separate review store.
- The intended review flow is `capture -> filed -> OS candidate -> inspect/explain -> approve|dismiss`, then optional explicit promotion through `khub os inbox triage`, `khub os task ...`, or `khub os decide`.
- 2026-04-10 live help matches this narrow wording: `khub os project` exposes `create|evidence|export-obsidian|list|show|update`, `khub os evidence` exposes `show|review`, and `khub os evidence review --help` shows `--action [approve|dismiss|explain]`. This remains candidate disposition only; it is not task/decision finalize.
- Gate framing for this slice is explicit:
  - live surface: `project evidence`, `evidence show`, and `evidence review` are shipped and should stay usable
  - supporting evidence: help/self-check plus the fresh 2026-04-10 review-only rerun (`3 passed in 0.97s` in this worktree)
  - blocker-worthy subset: still none for review semantics in the frozen narrow gate; if promotion is ever chosen, only the 3 review tests should move, not the broader review/list/show surface
- Operator workflow: `capture status -> capture list -> capture show -> recoverable orphan => capture requeue -> unrecoverable orphan => capture cleanup -> capture retry|capture-process once a queue packet exists`.
- Orphan remains a `flags/warnings/operatorAction` concern rather than a new status family.
- Recoverable orphan prefers requeue before retry/process.
- Unrecoverable orphan is handled through the runtime-local cleanup command.
- `capture requeue` is a schema-backed CLI surface.
- `capture cleanup` is a public CLI command for the runtime cleanup policy, but it remains dry-run-first and never becomes a canonical delete path.
- The internal cleanup helper now emits a strict `knowledge-hub.dinger.capture-cleanup.result.v1` payload with typed `delete` / `keep` entries for queue-backed captures, recoverable orphans, stale claims, and unrecoverable runtime junk so the current CLI surface and any future non-CLI consumers can reuse an explicit contract without reinterpreting operator semantics.
- Gate note: the shipped CLI surface now does include `khub os evidence review`, but the frozen approval gate still should not add it in this tranche. Revisit only if the review surface gets its own targeted verification that proves narrow candidate-disposition semantics and preserves the no-auto-promotion boundary.

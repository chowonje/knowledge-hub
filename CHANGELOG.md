# Changelog

## [Unreleased]

### Added

- Dinger facade MVP: `khub dinger ingest|ask|lint|recent` now provides a simpler user-facing knowledge surface over the existing `discover`, `crawl collect`, `ask`, and runtime-hygiene read models without changing canonical storage or Obsidian projection rules.
- `khub dinger file` now files ask results or direct notes into managed `KnowledgeOS/Dinger/Pages/*.md` projection pages and regenerates additive `KnowledgeOS/Dinger/Index.md` plus `KnowledgeOS/Dinger/Log.md` without introducing a new canonical store.
- `khub dinger capture` now provides a browser-friendly runtime intake surface for web clips (`sourceUrl`, `pageTitle`, `selectionText`, `capturedAt`, `client`, `tags`, optional raw pointers), persists queued packets under local runtime state instead of writing Dinger projection pages inline, returns `captureId` plus `accepted/queued`-style ack metadata, and keeps raw intake input-only until downstream filing and OS linkage.
- `khub dinger capture-process` now provides the local temp-runtime queue processor for queued capture packets. It writes inspectable sidecars under `dinger_capture_intake/runtime`, records adjacent runtime artifacts (`*.runtime.json`, `*.normalized.json`, `*.filed.json`, optional `*.os-linked.json`), advances packet lifecycle state through `captured -> normalized -> filed -> linked_to_os|failed`, skips already completed packets via capture-id/path idempotency, and uses the existing `dinger file` / `os capture` surfaces instead of introducing a new queue store or daemon.
- `khub dinger capture` now also exposes operator read-model subcommands: `capture list` derives `queued|processing|filed|linked_to_os|failed` from queue packets plus runtime artifacts, `capture show` returns the packet/state/artifact bundle for one capture, and `capture retry` replays only failed captures through the existing processor path without changing the filing or OS bridge contracts.
- Dinger capture orphan follow-up: when runtime artifacts remain but the queue packet is missing, `capture list` and `capture show` now surface the capture with explicit `packetPresent=false`, `flags/warnings`, and an operator action hint instead of hiding it, while `capture retry` still refuses to replay until the packet is restored or requeued.
- `khub dinger capture requeue` now restores only the canonical queue packet for orphan captures, auto-discovers exact packet snapshots for `--capture-id` when runtime hints or adjacent snapshot files exist, returns schema-backed `requeued|already_present|failed` semantics, refuses legacy orphans that have no recoverable snapshot, and leaves actual replay to the existing `capture retry` / `capture-process` paths.
- `knowledge-hub.dinger.capture-cleanup.result.v1` now defines the strict cleanup helper contract for runtime-only Dinger capture cleanup, including typed `delete` / `keep` entry variants for queue-backed captures, recoverable orphans, stale claims, and unrecoverable runtime junk without widening the cleanup policy or touching CLI wiring.
- `khub dinger capture cleanup` now surfaces the existing application cleanup helper as an operator CLI: default runs are dry-run previews, actual deletion requires `--apply --confirm`, JSON output preserves the helper's `delete|keep|errors|createdAt` shape with a schema envelope, and recoverable orphan captures stay visible under `keep` rather than being treated as deletable junk.
- `khub dinger capture status` now adds a lightweight operator aggregate surface over the existing read-model and cleanup helpers: it counts `queued|processing|filed|linked_to_os|failed` captures plus orphan/recoverable/retry-ready buckets from the current capture contract, counts stale/active claim files from the existing runtime cleanup scan, and emits immediate `requeue|retry|cleanup` next-action hints without adding a queue DB, new lifecycle states, or a new canonical store.
- Evidence-grounded Decision OS v1: `foundry-core` personal-foundry state now persists `decisions.json`, exposes append-only `decision add|list`, returns a derived `next` read model, and enriches project show/export payloads with decisions plus next-action summaries while keeping Obsidian projection-only.
- Knowledge↔OS Bridge v1: project-scoped Obsidian exports now live under `KnowledgeOS/Projects/<slug>/...`, a new read-only `khub os project evidence` surface exposes deterministic `projectEvidence`, and project show/export payloads now include bridge evidence derived from existing task/decision/inbox `SourceRef`s without changing canonical OS or knowledge storage.
- CLI-first Knowledge OS Workflow v1: `khub os` now adds workflow verbs (`capture`, `inbox triage`, `task start|block|complete|cancel`, and `decide`) as the canonical operator loop over the existing OS state, with schema-backed results and no new canonical store.
- `khub os evidence review` now adds a thin human-review surface over Dinger-backed `project evidence` candidates: `approve` reuses `inbox triage --resolve-only`, `dismiss` reuses `inbox resolve`, `explain` stays read-only, and candidate reason/explanation stays traceable to the filed Dinger page plus attached source refs without auto-promoting tasks or decisions.
- `khub os evidence show` now adds a read-only candidate detail surface, and `khub os project evidence` / `khub os evidence review --action explain` now render Dinger-backed candidates with clearer operator-facing `reuse/replay` context, matched open/resolved summaries, source refs, and next-action hints without changing the underlying OS bridge or review semantics.
